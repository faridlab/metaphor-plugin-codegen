# Architecture & Concepts

This document explains the architectural patterns and design principles behind the code generated by `metaphor-codegen`.

## Domain-Driven Design (DDD)

Metaphor Framework structures code around **Bounded Contexts** -- self-contained business domains with clear boundaries. Each bounded context is implemented as a **module** located at `libs/modules/<name>/`.

A module owns its entire vertical slice: from domain logic through database persistence to HTTP handlers. Modules communicate through well-defined contracts (Protocol Buffers), not by sharing internal types.

### Why DDD for Rust Microservices?

- **Clear ownership**: Each module has a single, well-defined responsibility
- **Independent deployability**: Modules can be extracted into separate services
- **Type safety**: Rust's type system enforces domain invariants at compile time
- **Proto contracts**: Cross-service communication uses Protocol Buffers, ensuring backward compatibility

## Clean Architecture Layers

Each module follows Clean Architecture with four layers. Dependencies point inward -- outer layers depend on inner layers, never the reverse.

```
┌─────────────────────────────────────────────┐
│              Presentation Layer              │  HTTP handlers, gRPC, CLI
│           src/presentation/                  │
├─────────────────────────────────────────────┤
│             Infrastructure Layer             │  Database, cache, messaging
│           src/infrastructure/                │
├─────────────────────────────────────────────┤
│              Application Layer               │  Commands, queries, use cases
│             src/application/                 │
├─────────────────────────────────────────────┤
│                Domain Layer                  │  Entities, value objects, events
│               src/domain/                    │  (no external dependencies)
└─────────────────────────────────────────────┘
```

### Domain Layer (`src/domain/`)

The innermost layer containing pure business logic with no external dependencies.

| Directory | Purpose | Generated By |
|-----------|---------|--------------|
| `entity/` | Core domain entities with identity | Schema-first approach |
| `value_object/` | Immutable objects defined by their attributes | `make value-object` |
| `event/` | Domain events (things that happened) | `make event` |
| `repository/` | Repository traits (interfaces only) | `make repository` |
| `service/` | Domain services (cross-entity logic) | `make service` |
| `specification/` | Business rules as composable predicates | `make spec` |

### Application Layer (`src/application/`)

Orchestrates domain objects to fulfill use cases. Implements the **CQRS pattern**.

| Directory | Purpose | Generated By |
|-----------|---------|--------------|
| `commands/` | Write operations (create, update, delete) | `make command` |
| `queries/` | Read operations (get, list, search) | `make query` |

### Infrastructure Layer (`src/infrastructure/`)

Implements interfaces defined by the domain layer using concrete technologies.

| Directory | Purpose | Generated By |
|-----------|---------|--------------|
| `persistence/` | Database repository implementations | `make repository` |

### Presentation Layer (`src/presentation/`)

Handles external communication -- HTTP endpoints, gRPC services, CLI commands.

| Directory | Purpose | Generated By |
|-----------|---------|--------------|
| `http/` | HTTP/REST handlers and routes | `make handler` |

## CQRS Pattern

The Command Query Responsibility Segregation (CQRS) pattern separates read and write operations:

### Commands (Write Side)

Commands represent intentions to change state. They are named as **imperative verbs**:

- `CreateUser` -- create a new user
- `UpdatePayment` -- modify an existing payment
- `DeleteOrder` -- remove an order
- `RestoreUser` -- restore a soft-deleted user

Generated location: `src/application/commands/<name_snake>.rs`

```bash
metaphor-codegen make command CreatePayment --module payments --entity Payment
```

### Queries (Read Side)

Queries retrieve data without side effects. They are named as **retrieval verbs**:

- `GetUser` -- fetch a single user by ID
- `ListPayments` -- retrieve a paginated list of payments
- `SearchOrders` -- search orders with filters

Generated location: `src/application/queries/<name_snake>.rs`

```bash
metaphor-codegen make query ListPayments --module payments --entity Payment
```

## Schema-First Approach

Metaphor uses a **schema-first** workflow where entity definitions live in YAML files that serve as the single source of truth. From these schemas, the framework generates:

- Protocol Buffer definitions (`.proto` files)
- Rust domain types
- SQL migration files
- API documentation

Schema files are located at:
```
libs/modules/<module>/schema/models/<entity>.model.yaml
```

Example schema:
```yaml
models:
  - name: Payment
    collection: payments
    fields:
      id:
        type: uuid
        attributes: ["@id", "@default(uuid)"]
      amount:
        type: float
        attributes: ["@required"]
      currency:
        type: string
        attributes: ["@required", "@length(3)"]
      status:
        type: string
        attributes: ["@default('pending')"]
```

> **Note:** The `make entity` command is deprecated in favor of the schema-first approach. Use `metaphor schema generate` from the `metaphor-plugin-schema` plugin instead.

## Proto-First Design

Protocol Buffers define the contract for cross-module and cross-service communication. Each module organizes its proto files under:

```
libs/modules/<module>/proto/domain/
  entity/          # Entity message definitions
  event/           # Domain event messages
  value_object/    # Value object messages
  repository/      # Repository service definitions
  service/         # Domain service definitions
  specification/   # Specification messages
  usecase/         # Use case service definitions
```

Proto files are generated alongside Rust code for domain events and value objects:

```bash
# Creates both proto/domain/event/payment_completed.proto
# and src/domain/event/payment_completed.rs
metaphor-codegen make event PaymentCompleted --module payments
```

## Key DDD Concepts

### Entity vs Value Object

| Concept | Identity | Mutability | Example | Command |
|---------|----------|------------|---------|---------|
| **Entity** | Has unique ID | Mutable over time | `User`, `Payment` | Schema-first |
| **Value Object** | Defined by attributes | Immutable | `Email`, `Money`, `Address` | `make value-object` |

**Use an Entity** when the object has a lifecycle and needs to be tracked by ID.  
**Use a Value Object** when the object is defined purely by its attributes and can be freely replaced.

### Domain Events

Domain events represent something that **happened** in the past. They are named in **past tense**:

- `UserCreated` -- a new user was created
- `PaymentProcessed` -- a payment was successfully processed
- `OrderCancelled` -- an order was cancelled

Each event generates both a `.proto` definition and a Rust handler.

### Specifications

Specifications encapsulate **business rules** as composable, reusable predicates:

- `UserIsActive` -- checks if a user account is active
- `PaymentIsValid` -- validates payment constraints
- `OrderCanBeCancelled` -- checks if an order is eligible for cancellation

Specifications can be composed with AND/OR logic for complex rules.

### Repository Pattern

The repository pattern separates data access from domain logic:

1. **Trait** (domain layer): Defines the interface at `src/domain/repository/`
2. **Implementation** (infrastructure layer): Concrete database logic at `src/infrastructure/persistence/`

This allows swapping database backends without changing domain code.

## Naming Conventions

| Element | Convention | Example |
|---------|------------|---------|
| Module names | snake_case | `payments`, `user_management` |
| Entity names | PascalCase | `User`, `PaymentTransaction` |
| Command names | PascalCase imperative | `CreateUser`, `UpdatePayment` |
| Query names | PascalCase retrieval | `GetUser`, `ListPayments` |
| Event names | PascalCase past tense | `UserCreated`, `PaymentProcessed` |
| Value object names | PascalCase | `Email`, `Money`, `Address` |
| Specification names | PascalCase predicate | `UserIsActive`, `PaymentIsValid` |
| App names | kebab-case | `my-service`, `payment-api` |
| File names | snake_case | `create_user.rs`, `payment_handler.rs` |
| Table names | snake_case plural | `users`, `payment_transactions` |
