# Migration Commands

Full PostgreSQL migration lifecycle management. Generates SQL from entity definitions, supports alter/diff operations, runs migrations via sqlx, and manages database seeders.

## Database Configuration

All migration commands that connect to the database resolve the connection URL using a centralized priority chain. See [Configuration](configuration.md) for details.

---

## migration generate

Generate a PostgreSQL migration for a single entity.

| Argument | Required | Default | Description |
|----------|:--------:|---------|-------------|
| `entity` | Yes | -- | Entity name in PascalCase (e.g., `Payment`) |
| `module` | Yes | -- | Target module name |
| `--force` | No | `false` | Overwrite existing migration file |

```bash
metaphor-codegen migration generate Payment payments

# Force overwrite
metaphor-codegen migration generate Payment payments --force
```

### How It Works

1. Resolves the migrations directory at `libs/modules/<module>/migrations/postgres/`
2. Determines the next migration number (e.g., `001`, `002`)
3. Converts the entity name to a table name (snake_case + pluralized)
4. Generates a SQL file with CREATE TABLE, indexes, triggers, and comments

### Generated File

```
libs/modules/payments/migrations/postgres/
  001_create_payments_table.sql
```

### Generated SQL Structure

The migration SQL includes:

```sql
-- CREATE TABLE with UUID primary key
CREATE TABLE IF NOT EXISTS payments (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    -- TODO: Add entity-specific fields
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW(),
    deleted_at TIMESTAMPTZ  -- soft delete
);

-- Indexes
CREATE INDEX idx_payments_deleted_at ON payments(deleted_at) WHERE deleted_at IS NULL;
CREATE INDEX idx_payments_created_at ON payments(created_at DESC);
CREATE INDEX idx_payments_deleted ON payments(deleted_at DESC) WHERE deleted_at IS NOT NULL;

-- Trigger for auto-updating updated_at
CREATE TRIGGER trigger_payments_updated_at
    BEFORE UPDATE ON payments
    FOR EACH ROW
    EXECUTE FUNCTION update_payments_updated_at();

-- Table and column comments
COMMENT ON TABLE payments IS 'Payment entity table - auto-generated by metaphor CLI';
```

> **Tip:** For migrations with entity fields automatically mapped to SQL types, use [`migration generate-all`](#migration-generate-all) instead, which parses your Rust entity structs.

## migration generate-all

Generate migrations for **all** entities found in a module's `entity_models.rs` file.

| Argument | Required | Default | Description |
|----------|:--------:|---------|-------------|
| `module` | Yes | -- | Target module name |
| `--force` | No | `false` | Overwrite existing migration files |

```bash
metaphor-codegen migration generate-all payments

# Force overwrite all
metaphor-codegen migration generate-all payments --force
```

### How It Works

1. Reads `libs/modules/<module>/src/infrastructure/persistence/postgresql/entity_models.rs`
2. Parses all `pub struct XyzEntity { ... }` definitions
3. Extracts field names and Rust types from `pub field: Type,` lines
4. Handles `Option<T>` for nullable columns
5. Generates numbered migration files for each entity

### Entity Parsing

The parser looks for struct declarations matching the pattern `pub struct <Name>Entity {`:

```rust
pub struct PaymentEntity {
    pub id: String,
    pub amount: f64,
    pub currency: String,
    pub status: String,
    pub user_id: String,
    pub notes: Option<String>,
    pub created_at: DateTime<Utc>,
    pub updated_at: DateTime<Utc>,
    pub deleted_at: Option<DateTime<Utc>>,
}
```

### Rust-to-PostgreSQL Type Mapping

| Rust Type | PostgreSQL Type | Notes |
|-----------|----------------|-------|
| `String` | `VARCHAR(255)` | |
| `i32`, `i64` | `BIGINT` | |
| `i16` | `SMALLINT` | |
| `u32`, `u64` | `BIGINT` | |
| `f32`, `f64` | `DOUBLE PRECISION` | |
| `bool` | `BOOLEAN` | Defaults to `false` |
| `DateTime<Utc>` | `TIMESTAMPTZ` | |
| `chrono::DateTime<chrono::Utc>` | `TIMESTAMPTZ` | |
| `Vec<String>` | `TEXT[]` | |
| `serde_json::Value` | `JSONB` | |
| `*_id` fields | `UUID` | Foreign key convention |
| Other types | `TEXT` | Fallback |

### Auto-Generated Indexes

Fields are automatically indexed when they match these patterns:

- Fields ending in `_id` (foreign keys)
- `email`, `username`, `name`, `key`, `token_hash` (common lookup fields)

### Standard Fields

These fields are added to every table automatically (skipped if present in the entity):

- `id` -- `UUID PRIMARY KEY DEFAULT gen_random_uuid()`
- `created_at` -- `TIMESTAMPTZ DEFAULT NOW()`
- `updated_at` -- `TIMESTAMPTZ DEFAULT NOW()` (with auto-update trigger)
- `deleted_at` -- `TIMESTAMPTZ` (soft delete, nullable)

## migration alter

Generate an ALTER TABLE migration to add new fields to an existing entity.

| Argument | Required | Default | Description |
|----------|:--------:|---------|-------------|
| `entity` | Yes | -- | Entity name in PascalCase |
| `module` | Yes | -- | Target module name |
| `-d`, `--description` | Yes | -- | Description of the change (used in filename) |

```bash
metaphor-codegen migration alter Payment payments --description "add_refund_fields"
```

### How It Works

1. Compares the current entity definition with the existing CREATE TABLE migration
2. Generates an ALTER TABLE migration for the new/changed fields
3. Creates a numbered migration file with the description in the filename

---

## migration diff

Compare entity fields with the existing migration to detect changes.

| Argument | Required | Default | Description |
|----------|:--------:|---------|-------------|
| `entity` | Yes | -- | Entity name in PascalCase |
| `module` | Yes | -- | Target module name |

```bash
metaphor-codegen migration diff Payment payments
```

Shows which fields exist in the entity definition but are missing from the migration, and vice versa. Useful for determining whether an alter migration is needed.

## migration run

Run pending migrations for a single module using sqlx.

| Argument | Required | Default | Description |
|----------|:--------:|---------|-------------|
| `-m`, `--module` | Yes | -- | Target module name |
| `--database-url` | No | auto | Database URL (falls back to config) |

```bash
# Using config-based database URL
metaphor-codegen migration run --module payments

# With explicit database URL
metaphor-codegen migration run --module payments --database-url "postgresql://user:pass@localhost:5432/mydb"
```

### Database URL Resolution

If `--database-url` is not provided, the URL is resolved from (in order):
1. `.env` file (`DATABASE_URL` or `POSTGRES_*` variables)
2. `apps/metaphor/config/application.yml`
3. `config/application.yml`

See [Configuration](configuration.md) for details.

## migration run-all

Run pending migrations for **all** registered modules.

| Argument | Required | Default | Description |
|----------|:--------:|---------|-------------|


## migration seed

Run database seeders for a module. Seeds populate the database with initial or test data.

| Argument | Required | Default | Description |
|----------|:--------:|---------|-------------|
| `-m`, `--module` | Yes | -- | Target module name |
| `-n`, `--name` | No | all | Specific seeder to run |
| `--force` | No | `false` | Re-run even if already applied |
| `--database-url` | No | auto | Database URL (falls back to config) |

```bash
# Run all seeders for a module
metaphor-codegen migration seed --module payments

# Run specific seeder
metaphor-codegen migration seed --module payments --name initial_payments

# Force re-run
metaphor-codegen migration seed --module payments --force
```

> **Note:** For more advanced seeding operations (create, revert, history), use the dedicated [seed commands](commands-seed.md).

## Migration File Naming

### CREATE TABLE Migrations

```
{number}_create_{table_name}_table.sql
```

Example: `001_create_payments_table.sql`

### ALTER TABLE Migrations

```
{number}_{description}.sql
```

Example: `002_add_refund_fields.sql`

### Make Command Migrations

```
{timestamp}_{name}.up.sql
{timestamp}_{name}.down.sql
```

Example: `20260414120000_create_payments_table.up.sql`

## Best Practices

1. **Use `generate-all` for initial setup** -- parses your entity structs and maps fields to SQL types automatically.

2. **Use `alter` for incremental changes** -- generates ALTER TABLE migrations instead of modifying the original CREATE TABLE.

3. **Always review generated SQL** -- especially the type mappings and indexes. Adjust `VARCHAR(255)` lengths and add custom constraints as needed.

4. **Run `diff` before `alter`** -- verify what changes are detected before generating an alter migration.

5. **Use `--force` sparingly** -- overwriting migrations that have already been applied to databases can cause inconsistencies.

6. **Test migrations in development first** -- use `migration run` with a development database before applying to production.
